1. 目的、范围与设计结论
本文档规定 ChinaTextbook v1.1 如何从唯一官方仓库发现教材、把用户选中的“书”解析为精确源文件、按需获取文件、合并与验证 PDF、检测教材变化，并提示软件新版本。它不重新定义 2.3 的数据字段或 2.4 的状态机，而是说明这些模型和状态在 Git 与网络 I/O 中如何落地。
1.1 纳入范围
官方仓库发现、首次克隆、fetch、工作区校验与修复。
基于固定 commit 的目录快照、书目聚合和文件大小信息。
selected_books 到 selected_files 的精确映射及下载计划冻结。
部分克隆、精确文件 sparse-checkout、按需 blob 获取与工作区验证。
下载暂存、分卷合并、SHA256 验证、原子发布和状态持久化。
教材仓库更新检测、差异分类、已选教材更新。
通过 GitHub Releases 进行软件版本提示；v1.1 不自动覆盖程序。
1.2 不纳入范围
多镜像测速、Gitee 或用户自定义下载源。
Git 提交、推送、合并、rebase、分支开发等写入上游仓库的操作。
静默自动升级 EXE 或 Python 包。
基于文件名猜测出版社、ISBN、发布日期等不可靠元数据。

2.2 单源信任模型
软件只接受预置的 HTTPS 官方 URL。任何已有工作区在使用前都必须验证 origin URL 的标准化结果与预置 URL 一致；不一致时进入安全模式，不自动修改到“看起来相近”的地址。重定向只能由 Git/HTTPS 层处理，日志中记录最终主机但不记录认证参数。
禁止：不得保留 v1.0 的镜像测速、最快源选择、Gitee 分支或 --use-gitee 等入口。配置文件中出现旧字段时迁移器应忽略并记录迁移说明。
2.3 Git 特性现实约束
Git 的 partial clone 可在 clone/fetch 阶段省略不需要的对象，缺失 blob 在实际需要时按需获取。Sparse checkout 控制工作树出现哪些路径。二者作用不同，v1.1 同时使用：partial clone 降低初始对象传输，sparse checkout 限制工作树。
精确文件选择要求 non-cone pattern 模式；Git 官方文档已将 non-cone 标为弃用并建议使用 cone mode。由于 cone mode 只能以目录为选择单位，v1.1 将 non-cone 作为受控例外：模式由程序生成、通过 stdin 传递、工作区只读、禁止 merge/rebase/pull、禁用 sparse-index、限制模式数量，并在每次检出后进行集合级验证。该风险必须进入 ADR 和回归测试。

3.3 Commit 身份
任何目录快照、Book、SourceFile、DownloadPlan 和 UpdateSummary 都必须携带 commit SHA。计划开始后，即使远端 HEAD 更新，当前任务仍在原 commit 上执行；新 HEAD 只能触发新的目录刷新或更新任务，不能“热切换”正在运行的下载。
4. Git 能力检测和仓库初始化
4.1 启动能力检测
1. 执行 git --version，解析主、次版本；低于 2.27 阻止下载和更新。
2. 检查 git 可执行文件路径，诊断报告记录路径和版本。
3. 验证工作区父目录可写、剩余空间、路径长度与临时文件原子替换能力。
4. 获取进程锁；第二实例不得进入写模式。
5. 如果工作区存在，验证 .git、origin URL、partial clone 配置和当前 HEAD。
4.2 默认分支发现
默认分支不得硬编码为 main 或 master。初始化时使用 Git 协议查询远端符号引用：
git ls-remote --symref https://github.com/TapXWorld/ChinaTextbook.git HEAD
解析 ref: refs/heads/<branch> HEAD 与后续 HEAD SHA。解析失败时可使用 GitHub repository metadata 作为后备；两者均失败则不创建仓库。
4.3 首次克隆
git clone \  --filter=blob:none \  --no-checkout \  --single-branch \  --branch <default_branch> \  https://github.com/TapXWorld/ChinaTextbook.git \  <workspace>
不使用 shell=True；命令及参数以数组传递。blob:none 使 blob 在真正需要时才按需获取。v1.1 不默认使用 shallow clone，以免更新比较、恢复和历史 commit 可达性变复杂；后续性能数据证明必要时可单独评审。
图 2  首次初始化流程
4.4 初始化后验证
git rev-parse --is-inside-work-tree 必须为 true。
git remote get-url origin 的规范化结果必须等于官方 URL。
extensions.partialClone 或 remote.origin.promisor / partialclonefilter 与预期一致。
HEAD、远端默认分支和 catalog commit 均为 40/64 位十六进制 Git object ID 的可接受形式；实现不得假定永远是 SHA-1 长度。
工作区内不允许用户自定义 hook、submodule 自动初始化或额外 remote。
5. 目录快照与书目索引刷新
5.1 快照生成原则

普通 *.pdf 映射为一个 Book，source_files 只有一个。

书目聚合后计算 catalog fingerprint，供缓存、迁移和测试使用。

6. 精确文件选择与 sparse-checkout 设计
6.1 选择映射
SelectionService 以 selected_books 为用户状态，以 selected_files 为执行状态。每次启动下载前都必须从当前 catalog snapshot 重新解析 Book → SourceFile，排序去重后生成 selection_fingerprint。旧状态中的目录选择不参与此过程。

6.3 受控 non-cone 策略
# patterns 由程序生成并通过 stdin 传入# 每一条都是根锚定、转义后的“精确文件”模式/小学/数学/一年级/教材A.pdf/初中/物理/八年级/教材B.pdf.1/初中/物理/八年级/教材B.pdf.2# 执行 git -C <workspace> sparse-checkout set \   --no-cone --no-sparse-index --stdin
这里的换行仅用于说明；实际调用不包含 shell 续行。由于 non-cone 已被 Git 官方弃用，实现必须将其封装在 ExactMaterializer 接口后，禁止其他模块依赖具体命令。
受控风险：non-cone 的性能约为模式数 × index 路径数，且模式语义容易出错。v1.1 限定一次计划最多 2,000 个源文件；超过上限拆批或拒绝。每次 set 之后必须运行规则验证和实际文件集合验证。
6.4 检出顺序
1. 确认工作区无未跟踪业务文件、无用户修改、无冲突和锁文件。
2. fetch 目标 commit 与 tree；不直接 pull。
3. 写入精确 pattern，并使用 check-rules 验证所有 selected_files 均匹配。
4. checkout --detach <plan_commit>，使缺失 blob 通过 promisor remote 按需获取。
5. 枚举工作区中教材候选文件；实际集合必须等于计划集合，不允许同目录多下载其他 PDF。
6. 对每个文件核对路径、blob identity（可验证时）和预期大小；不一致则失败。

6.5 可替换实现
ExactMaterializer 接口应允许未来将实现替换为“按 commit:path 逐 blob 物化到暂存区”，不改变 Book、DownloadPlan、Task 或 UI。触发替换评审的条件包括：Git 删除 non-cone 支持、目标仓库规模导致性能不可接受、或跨平台规则匹配出现不可消除差异。

7.2 磁盘空间预算
预检按最坏并行存在计算，不只看下载体积。建议公式：
required = missing_source_bytes + merge_temp_bytes + publish_temp_bytes + state_margin + 10% safety_margin
完整 PDF 的 merge_temp_bytes 为 0；分卷教材按分卷总和估算合并临时文件。size 未知时，不能显示伪精确值；采用保守默认或要求用户确认，并在执行中持续复核剩余空间。

8. 下载事务和文件发布
8.1 事务边界
一个下载任务可以包含多本书，但每本 Book 具有独立的处理结果；任务总体完成要求所有计划 Book 均进入 ready 或被明确记录为失败/跳过。默认采用“逐书提交”：一本书验证和发布成功后立即持久化，后续书失败不回滚已成功书。
8.2 标准流程
1. 冻结 plan，写 Task=pending；Worker 启动后转 running/preflight。
2. fetch 目标 commit；必要时刷新 promisor objects。
3. 应用 exact sparse rules 并 checkout 固定 commit。
4. 把精确源文件复制/硬链接（仅确认同卷安全时）到 task staging/source；记录 staging manifest。
5. 普通 PDF 直接进入验证准备；分卷 PDF 按 2.3 中 parts 顺序合并到 staging/merged/<name>.tmp。
6. 计算 SHA256，并执行 PDF 基础可读性/尾部标记检查；不得仅以扩展名判断。
7. 将最终候选写到 output 同目录的临时文件或同卷临时名，fsync 后 os.replace 原子发布。
8. 在锁内保存 Group、Verification、Book local status 和 Task progress。
9. 只有状态保存成功后，Book 才向 UI 发布 ready；任务最后进入 persist/done/completed。
图 4  下载、处理、验证与发布事务
8.3 旧输出保护
目标已存在且状态为 ready 时，新任务先写新的临时候选，不覆盖旧文件。
新候选 SHA256、PDF 验证或状态保存失败，旧成品保持不动。
原子 replace 成功但状态保存失败时，启动恢复器根据文件真实哈希和计划证据校正，不直接宣布完成。
用户显式“重新下载”不等于先删除；替换仍采用先验证新候选、后原子发布。

9.2 速度与 ETA
速度使用滑动窗口，过滤启动尖峰；ETA 仅在连续获得足够样本且总量已知时显示。Git 的 receiving objects 与按需 blob fetch 可能分多次发生，不能把一次 Git 百分比等同于整个任务百分比。

10.2 超时
区分连接超时、无输出超时和总任务上限；大文件下载不应因总耗时长而误杀，只在长期无进展且无磁盘写入时判断卡死。
Git 子进程 stdout/stderr 必须持续消费，防止管道填满。
超时先发送温和终止，等待宽限期，再终止子进程树；不得杀死 Python UI 线程。
超时后检查 .git/index.lock、shallow.lock、临时 pack 等材料，只有证明安全才自动清理。
10.3 用户取消
取消遵循 2.4：running → cancelling，停止领取新 Book，在 Git 子进程结束、文件块边界、合并输出关闭、哈希轮次结束或原子事务边界处转 cancelled。已经验证并发布的 Book 不因取消其他 Book 而删除。
10.4 崩溃恢复
1. 启动时取得锁并读取 state；旧 pending/running/cancelling 任务改为 interrupted。
2. 扫描 staging manifest，只处理带 task_id、plan fingerprint 和安全边界标记的目录。
3. 如果最终输出已存在，按真实 SHA256 与计划证据判断 ready、stale 或 failed。
4. 恢复操作创建新 task_id；旧任务只作历史记录。
5. 无法判定临时文件归属时隔离或保留，不自动删除。

11. PDF 合并、校验与清理的衔接
11.1 合并输入不可变
Merger 只接受 DownloadPlan 固定的 parts 列表，不自行扫描目录猜测分卷。每个 part 必须在 staging manifest 中记录源 Git path、blob SHA、大小和本地路径。
11.2 验证顺序
1. 验证源集合与 parts 连续性。
2. 流式合并到 .tmp，不在 output 直接拼接。
3. 检查候选至少具有 PDF 头部/尾部和可读取的基本结构；具体库由实现选型决定。
4. 计算候选 SHA256 和大小。
5. 原子发布后可选重读 SHA256；高安全模式必须重读。
6. 保存 Group hash、size、parts、verified_at 和 source commit。
11.3 clean 规则
clean 仅作用于当前 task staging 中已登记的分卷，不作用于 Git 工作区以外的未知文件。
只有最终输出已验证且状态保存成功，才允许删除 staging 分卷。
--skip-verify 时不得执行 clean；UI 必须显示风险。
Git sparse-checkout 自己移除工作区文件不等同于 clean，不影响输出和历史 Group 记录。
12. 教材仓库更新检测
12.1 检查触发
用户按 F9 主动检查。
启动后后台延迟检查；不得阻塞主界面。
上次成功检查超过可配置间隔时检查；默认 24 小时。
下载开始前只获取远端 HEAD，发现目录缓存落后时提示刷新，不强制改变当前计划。
12.2 Commit 比较
1. 读取 local_catalog_commit。
2. 通过 git ls-remote 或轻量 fetch 获取 remote_head_commit。
3. 相同则无更新。
4. 不同则构建 remote catalog snapshot。
5. 优先以 Book 快照进行语义比较；GitHub compare commits API 用作提交和文件变化辅助信息，不作为完整目录权威来源。
GitHub compare endpoint 可提供 added、removed、modified、renamed 等文件状态，但大规模比较存在提交/分页限制；因此产品更新摘要最终以两个完整 catalog snapshot 的差异为准。

15. GitHub API、缓存与限流
15.1 API 客户端统一要求
所有 REST 调用经 GitHubApiClient，不允许模块自行拼 URL。
固定 Accept、API version、User-Agent；支持 ETag/If-None-Match。

解析 JSON 前检查 status、Content-Type 和体积；错误正文截断并脱敏。
404 latest release 解释为“尚无稳定 Release”，不是应用故障。

15.3 限流
读取响应头中的 remaining、reset、retry-after 等信息。达到限流后不进行高频重试：目录刷新优先退回本地 Git tree；软件更新显示“稍后再检查”；状态栏可以显示预计可重试时间。应用不要求用户提供 token 才能完成核心下载。

17. 安全、隐私与完整性边界
17.1 路径与文件安全
所有输出路径经 resolve 后必须仍位于 output 根目录；拒绝 symlink/junction 逃逸。
仓库中的 symlink、submodule 和非普通 blob 不作为教材源。
临时文件名由 task_id 与随机值生成，不使用不可信书名直接构造。
Windows 保留名、尾随点/空格、路径长度和大小写冲突必须在计划期检测。
删除只允许针对 manifest 中登记且位于 staging 根内的路径。
17.2 网络和供应链
固定官方 GitHub 主机和仓库；不执行仓库中的脚本、hook、LFS smudge 或 submodule。
建议设置 GIT_LFS_SKIP_SMUDGE=1、禁用 credential prompt，并清理可影响行为的危险环境变量。
软件更新只打开官方 Release 页面；v1.1 不下载和执行更新资产。
API token（未来可选）不得进入命令行、日志、状态文件或诊断包。
17.3 完整性证据优先级
1. 最终文件的真实 SHA256 和大小。
2. 计划 commit、精确 source path、blob SHA 和 parts 清单。
3. 原子状态记录中的 Group/Verification。
4. Catalog cache 和 UI 显示。
低级证据不能覆盖高级证据。例如 UI 显示“完成”但文件 SHA256 不符，应以真实文件为准并转 stale/failed。

附录 B. 稀疏规则示例
B.1 普通文件
Git path:小学/数学/一年级/人教版数学一年级上册.pdf规则（概念表示）：/小学/数学/一年级/人教版数学一年级上册.pdf
B.2 特殊字符转义
Git path:初中/物理/八年级/[实验]力与运动!.pdf.1根锚定 exact pattern 必须转义 pattern 元字符：/初中/物理/八年级/\[实验\]力与运动\!.pdf.1
转义函数必须由测试向量验证，不应由 UI 或调用方手工处理。由于 Git non-cone 使用 gitignore-style patterns，开发人员必须以实际 Git check-rules 结果作为最终判据。
B.3 集合后置条件
expected = sorted(plan.selected_files.paths)matched  = sorted(git_sparse_check_rules(expected))actual   = sorted(scan_tracked_pdf_sources(workspace))assert matched == expectedassert actual == expected

[6] GitHub REST API - Releases — 最新稳定 Release 定义与公开资源访问。

参考资料的在线行为可能随 Git/GitHub 版本变化。实现应以 CI 中固定版本矩阵和契约测试结果为准，并在升级依赖前更新本文档。
