1. 信息架构
v1.1 的信息架构以“教材发现、选择、获取、验证、维护”五个阶段组织。主界面承担发现和选择；任务中心承担获取；详情与日志承担验证和诊断；更新中心承担维护。

2. 内容分类与书目模型
2.1 导航顺序
固定导航：学段 → 科目 → 年级/册次 → 具体教材。科目位于年级之前。
程序从 Git 树路径、文件名和目录层级推断分类，但不得硬编码仅适用于小学/初中/高中。对于大学、五四学制、必修/选择性必修等结构，分类器应保留真实目录名称并映射到最接近的层级。
2.2 Book 聚合规则
输入文件

教材A（普通 PDF）
教材B.pdf.1 + 教材B.pdf.2 + 教材B.pdf.3
教材B（3 个分卷）
BOOK.PDF.01 + BOOK.PDF.02
BOOK（保留原文件名，卷号按整数排序）
Book ID 使用不含分卷后缀的仓库相对路径。source_files 保存实际文件列表；part_numbers 保存卷号；estimated_size 为所有源文件大小之和。

4. 核心用户流程
4.1 首次运行
启动并获取单实例锁。
检测 Git、GitHub、磁盘、工作区和状态文件。
若无工作区，后台执行 GitHub 官方仓库轻量克隆。
读取 Git 树并构建书目索引。
进入主浏览器，默认展开常用学段，不自动下载。
用户逐级导航并用 Space 选择教材。
F5 打开下载确认摘要，确认后执行文件级 sparse-checkout。
下载完成后按组自动合并、验证和更新状态。

4.3 搜索流程
用户按 /，焦点进入搜索输入框。
输入多个关键词，系统对 title、路径、学段、科目、年级做 AND 匹配。
结果实时刷新，↑↓移动，Space 直接切换选择。
Enter 定位到主树中的教材并关闭搜索；Esc 取消搜索并恢复原焦点。
4.4 更新流程
后台比较本地 HEAD 与 origin 默认分支。
无变化时仅更新时间戳。
有变化时计算 diff，将受影响 groups 标记 stale，保留历史 parts。
更新中心展示新增、修改、删除/结构变化。
用户选择“更新已选教材”后执行 fetch/fast-forward、重新检出、恢复和核验。
发现上游删卷或分卷结构缩短时，保留现有 PDF 并要求人工核对。
