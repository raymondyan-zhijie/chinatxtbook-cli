# ChinaTextbook 下载与合并工具 v1.0 正式版

适用于 `ChinaTextbook` 仓库中 **小学 / 初中 / 高中**（也支持 大学、五•四学制 等其他顶层目录）的按需下载，以及 `.pdf.1`、`.pdf.2` 这类分卷文件的自动合并。

脚本文件名：`textbook.py`

> 版本号说明：历经六轮安全审核（开发号 v4.0–v4.3 RC）后，统一以 **v1.0** 作为首个正式发布版本；旧开发版状态文件（4.1–4.3）兼容沿用。下文历史章节保留开发号以便对照审核记录。

> 提示：教育部「国家中小学智慧教育平台」(basic.smartedu.cn) 提供官方免费电子教材，是最权威的获取渠道。本工具仅针对社区仓库的批量下载与分卷合并场景，请自行确认使用场景的合规性。

## v1.0 相对 v4.3 的变化（第六轮审核，发布阻断缺陷修复）

第六轮审核发现一个会导致**完整 PDF 被残缺版本静默覆盖**的阻断问题，修复后发布 v1.0 正式版：

| 级别 | 问题（v4.3） | v1.0 处理 |
|---|---|---|
| P0 | **上游删除最后一卷后，`--update` 生成残缺 PDF 并报成功**：删卷使记录被 diff 失效（直接删除），随后按"变短的"Git 清单重合并——Git 树只描述"现在有什么"，无法描述"原本应有什么"，删卷后的清单依然"连续"（实测 `AAAABBBB` 被覆盖成 `AAAA` 且日志"全部完成!"） | 记录失效改为 **stale（过期）标记**，保留历史 `parts` 集合与 sha256；合并前比对历史分卷集合与新清单，**新清单缺少历史上存在过的卷即拒绝覆盖**（保留现有 PDF，报错如："检测到上游删除了原有分卷 [2]，无法确认新清单完整性；已保留现有 PDF，拒绝自动覆盖"）。确认上游变更合法时，删除输出文件重跑即按新清单重建 |
| P0 | 同理，diff 失败时"清空全部记录"会丢失历史分卷集合，使删卷不可检测 | diff 失败/HEAD 缺失时改为**全量 stale 标记**（同样保留 parts），而非清空；stale 记录不作为任何跳过证据 |
| P0 | 清单仅有 `.pdf.1` 的**单卷组**无从判断完整性（首次运行无历史可比） | **默认拒绝合并**：分卷的存在本身意味着原文件超过单文件体积限制，正常拆分不会只有 1 卷 |
| — | 版本标识不统一（文件名 v4.4 vs `VERSION="4.3"`） | 统一为发布版 **v1.0**：`VERSION = "1.0"`，`COMPATIBLE_STATE_VERSIONS = {"4.1", "4.2", "4.3", "1.0"}` |

配套机制：

- 旧版（v4.2/v4.3）记录无 `parts` 字段：哈希核对通过的跳过会自动**回填**当前清单分卷集合（此时记录非 stale，上游自合并后未变，回填是安全的）；若在回填前就遇上上游变更被标记 stale，则 fail-closed 拒绝自动覆盖，提示人工核对
- `--status` / `--report` 中 stale 记录标注为【过期，待重新核对】

> 诚实说明（本轮撤回的承诺）：此前 README 中"即使最后一卷整卷缺失也能发现"的表述**不准确**——它只对"本地缺卷但 Git 清单完整"成立；若是**上游清单本身被删去末卷**，v4.3 及之前版本会拼出残缺 PDF 并报成功。v1.0 起，对**有历史记录**的组可检测并拒绝；对**首次见到**的组（无历史可比），若末卷在本工具首次运行前就已被上游删除，除单卷组外**无任何本地手段可检测**，详见"已知限制"。

## v4.3 相对 v4.2 RC2 的变化（发布阻断缺陷修复）

发布前测试发现一个会导致**旧 PDF 被错误保留**的阻断问题，修复后按 v4.3 发布：

| 级别 | 问题（v4.2 RC2） | v4.3 处理 |
|---|---|---|
| P0 | **切换到领先的同源镜像后，旧哈希缓存未失效**：已 `--clean` 的组在切源后只拿"现有 PDF"比对"旧记录哈希"，旧 PDF 当然匹配旧记录，日志显示"哈希核对一致"并跳过，上游新内容被静默丢弃（实测应为 `CCCCBBBB` 的输出仍是 `AAAABBBB`） | 切换前后比对 HEAD；发生变化时用 `git diff --name-only OLD NEW` 按 `--update` 的同一逻辑**精确失效上游已变化的组记录**（未变化组保留，免重合并）；diff 无法计算时清空全部 groups（fail-closed；v1.0 起改为全量 stale 标记以保留删卷检测能力，见上方 v1.0 说明）。失效组的缺失分卷在合并阶段自动 `git restore` 后重新合并 |
| — | 版本标识不统一（文件名与 VERSION/兼容集合/文档不一致） | 统一为 v4.3：`VERSION = "4.3"`，`COMPATIBLE_STATE_VERSIONS = {"4.1", "4.2", "4.3"}`，docstring 与 README 同步 |

> 诚实说明：此前版本中"切换源后内容如已变化，合并阶段哈希核对会自行发现"的表述**不成立**：对已清理分卷的组，核对基准是旧记录自身，构成自证循环。v4.3 改为在切换时主动失效受影响记录，该承诺自 v4.3 起才真正成立。

## v4.2 相对 RC 的变化（第四轮审核修复）

第四轮审核在 v4.2 RC 中发现了 fail-closed 原则的最后一个缺口及若干工程问题，v4.2（RC2）逐项修复：

| 级别 | 问题（RC） | v4.2 处理 |
|---|---|---|
| P0 | **Git 清单外的本地分卷组仍是 fail-open**：目录里孤立的 `x.pdf.1` 会被当作完整组合并成 `x.pdf`，`--clean` 随后删掉唯一原件 | **默认拒绝**：不在 Git 树清单中的分卷组一律报错拒绝合并（无法确认末卷完整）；处理目录集只来自清单，工作区扫描仅用于发现并提示异常，不再新增可合并目录 |
| P1 | 上游删除/重命名已选目录后 `--update` 静默"成功"（检出阶段被旧标记短路） | 实际发生更新后强制重新检出并逐个校验已选路径在 Git 树中的存在性，缺失时明确报错并提示 `--reselect` |
| P1 | 部分错误日志直接输出原始 git stderr，可能泄露 URL 中的凭据 | 新增 `safe_error()` 统一脱敏 + 截断，所有外部错误入日志前必经过此函数 |
| P1 | 仓库切换非事务化：被拒绝的切换也会永久清空已完成组的 sha256 记录；分支同步失败不回滚 origin | 切换前快照 origin 与完整状态，任一步失败完整回滚；全部成功后才失效派生缓存（注：当时"保留哈希记录、内容变化由哈希核对自行发现"的设计对已清理分卷的组不成立，v4.3 已改为按 HEAD diff 主动失效，见上方 v4.3 说明） |
| P1 | `--skip-verify` 下对"无历史记录但同大小"的现有 PDF 按 size-only 信任跳过，且日志统一写"已验证一致"混淆核对方式 | 删除该信任跳过分支（无可信记录直接重建）；日志严格区分"哈希核对一致，跳过"与"按历史大小快速跳过（未哈希核对）" |
| P2 | 哈希核对通过的跳过组在 `--clean` 时不清理残余分卷（如上次清理被中断） | 跳过结果携带残余分卷清单，`--clean` 时顺带删除 |
| P2 | `--report` 写文件但不持锁、非原子写 | `--report` 需获取单实例锁；报告改为临时文件 + `os.replace` 原子替换（`--status` 仍无锁只读） |
| P2 | 陈旧锁回收存在"两进程都删除+创建"的竞争窗口 | 回收改为原子 `rename`，同一时刻只有一个进程能回收成功，其余重试 `O_EXCL` 创建 |
| P2 | `os.replace` 后未 fsync 父目录，极端断电场景下 rename 元数据可能丢失 | POSIX 下合并产物落盘后额外 fsync 父目录，再允许后续删除分卷 |

## v4.2 相对 v4.1 的关键变化（第三轮审核修复）

第三轮审核发现 v4.1 的若干承诺**并未完全兑现**，v4.2 逐项修复并撤回/重述如下（诚实说明，先撤回再兑现）：

| v4.1 中的表述 | 实际情况（v4.1） | v4.2 处理 |
|---|---|---|
| "每次运行都会核对，被篡改输出自动自愈" | 校验模式下仍存在"记录 + 大小一致"快速跳过，同大小篡改不会被发现；已清理组只比记录大小、从不重算 sha256 | **已修复**：校验模式（默认）下分卷齐全一律完整哈希比对；已清理组用记录 sha256 重算现有 PDF 核对，不一致自动恢复重合并。按大小快速跳过仅在显式 `--skip-verify` 下允许 |
| 已清理组可用"上游元数据大小核对" | API 文件大小之和被当作完整性证据，状态丢失/拷贝/升级后同大小错误 PDF 会被误判为完成 | **已移除**：API 大小只用于磁盘估算与展示，永远不作为内容正确的证据；无可信历史哈希一律恢复分卷重新哈希核对 |
| 清单来自 Git 树，可发现末卷缺失 | Git 树读取失败时静默降级为工作区推断（fail-open），孤立的 `book.pdf.1` 仍会被合并 | **已修复**：清单读取失败即停止处理（fail-closed），不再降级推断 |
| "`--clean` 时校验仍强制执行" | `--clean --skip-verify` 组合下，新合并输出的重读校验实际被跳过，分卷随后被删除 | **已修复**：`--clean` 强制新输出重读校验与哈希验证，`--skip-verify` 无法关闭 |
| "切换仓库源会同步切换内容" | `--repo` 切换只改 `origin` 并 fetch，本地分支与工作区仍是旧仓库内容（显示新源、处理旧源） | **已修复**：切换后检查历史兼容性——同源镜像自动快进同步到新源分支；无关仓库**拒绝在原工作区切换**并回滚，提示到新目录重跑 |

其他修复：

| 级别 | 修复内容 |
|---|---|
| P1 | 旧临时文件清理移到线程池启动前按目录一次性执行，同目录多组并发合并不再互删对方正在写入的 `.pdf.tmp` |
| P2 | `groups` 作为跨选择的持久缓存保留（回选旧目录免重合并），但 `--status` / `--report` 只统计**当前选择范围内**的组，并注明历史缓存条数 |
| P2 | 分卷扫描限定在当前选定路径内并排除 `.git`，不再遍历整个工作区 |
| P2 | URL 脱敏扩展到查询参数（`?access_token=…`、`?private_token=…` 等），日志/状态/报告全覆盖 |
| P2 | 新增单实例锁 `china_textbook.lock`（记录 PID，进程退出后的陈旧锁自动回收），同目录并发运行会被明确拒绝 |

**状态文件**：v4.1–v4.3 状态与 v1.0 完全兼容，升级后直接沿用（保留哈希记录，无 parts 字段的旧记录在首次哈希核对通过时自动回填）；更早版本自动备份后重建。

## 环境要求

1. Python 3.9 或更高版本
2. Git 2.27 或更高版本（需支持 `sparse-checkout reapply`；2.42+ 可获得更严格的懒加载防护）
3. 可访问 GitHub 或 Gitee 的网络环境

支持 Windows 10/11（PowerShell / CMD / Windows Terminal）、macOS、Linux。

## 快速开始

将脚本放到一个独立目录，例如：

```powershell
mkdir D:\ChinaTextbookTool
cd D:\ChinaTextbookTool
python textbook.py
```

首次运行流程：

1. 测速并选择更快的仓库源（GitHub / Gitee）
2. 轻量克隆仓库骨架（只有提交与目录树，不下载 PDF）
3. 通过 API 一次性获取所有目录的精确大小（不产生文件下载）
4. 列出目录供交互式选择，**每项直接标注体积**
5. 确认后按预估体积校验磁盘空间，再执行稀疏检出下载
6. 依据 Git 树清单核对并合并分卷 PDF（原子写入 + 重读一致性校验）

## 命令一览

| 命令 | 说明 |
|---|---|
| `python textbook.py` | 正常运行（含断点恢复；默认对所有组做哈希核对） |
| `python textbook.py --list` | 仅列出目录树与大小，不下载 |
| `python textbook.py --reselect` | 重新选择下载目录 |
| `python textbook.py --status` | 查看各阶段进度、失败组及原因（只统计当前选择范围） |
| `python textbook.py --dry-run` | 预览核对结果与待合并清单（含缺卷/需恢复检测），不执行；可与 `--update` 组合 |
| `python textbook.py --clean` | 合并且**哈希验证通过**后删除分卷文件（可事后单独执行；重读校验强制开启） |
| `python textbook.py --update` | 拉取上游更新，仅重新处理有变化的组；被 `--clean` 清理且上游有变化的分卷会自动恢复 |
| `python textbook.py --use-gitee` | 强制使用 Gitee 镜像 |
| `python textbook.py --repo URL` | 使用自定义仓库地址（切换时做历史兼容性检查，见下文） |
| `python textbook.py --dirs 小学,大学` | 自定义顶层目录（仓库内还有 大学、初中（五•四学制） 等） |
| `python textbook.py --workers 4` | 并发线程数（1-8，机械硬盘建议 1） |
| `python textbook.py --skip-verify` | 跳过哈希核对，允许按**历史记录**大小快速跳过；无记录的同大小文件不会被信任，直接重建（**信任换速度**；`--clean` 时校验仍强制） |
| `python textbook.py --report` | 生成 Markdown 运行报告（只统计当前选择范围） |
| `python textbook.py --retry` | （兼容保留）每次运行都会自动核对并重试未完成的组 |

## 交互选择说明

程序会先列出可选目录及体积，例如：

```text
【小学】
[  1]   一年级  (~1.20 GB)
[  2]   二年级  (~1.15 GB)

【初中】
[  3]   七年级  (~980.0 MB)

【高中】
[  4]   数学  (~1.35 GB)
```

可输入的选择方式：

- `all`：全选
- `1,3`：选择多个编号（支持中文逗号）
- `1-3,5`：选择范围 + 单项
- `小学`：选择所有【小学】类别下的目录
- `初中,高中`：选择多个类别
- `数学`：按目录名子串匹配
- `skip`：退出，不下载

选择后程序立即给出预计下载总量（来自 API 元数据缓存，秒级完成、零下载），确认后才开始下载。

├─ china_textbook_report.md     # 运行报告
└─ ChinaTextbook_Workspace      # Git 工作目录与下载内容
   ├─ .git

## 分卷 PDF 合并说明

脚本自动扫描 `某教材.pdf.1`、`某教材.pdf.2` … 并合并为 `某教材.pdf`。

### 合并策略

- **清单来源**：以 `git ls-tree` 的仓库树为权威清单，而非仅凭工作区现有文件推断——**本地**缺卷（含末卷）只要 Git 清单完整就能发现并自动恢复。**清单读取失败时直接停止处理（fail-closed），绝不降级为工作区推断**
- **历史集合比对（v1.0）**：Git 树只描述"现在有什么"，无法描述"原本应有什么"；因此每组的历史分卷集合（`parts`）随哈希一同记录，上游变更后新清单缺少历史上存在过的卷即拒绝覆盖（保留现有 PDF）
- **单卷组拒绝（v1.0）**：清单仅有 `.pdf.1` 的组默认拒绝合并（无法证明不存在被删除的后续分卷）
- **清单外拒绝**：处理目录集只来自清单；不在清单中的本地分卷组（包括看起来"连续"的孤卷 `x.pdf.1`）一律拒绝自动合并与清理，扫描发现清单外目录含分卷时仅提示不处理
- **连续性校验**：分卷必须从 `.1` 开始且连续。缺卷会明确报错并指出缺失卷号；上游清单本身缺卷会单独标为上游数据问题。注意：连续性校验只能发现**中间**缺卷；末卷被上游删除后清单依然"连续"，需依靠历史集合比对与单卷组拒绝才能拦截（首次见到的多卷组无法拦截，见"已知限制"）
- **重复卷号检测**：`.pdf.1` 与 `.pdf.01` 共存视为异常并报错（`.pdf.01` 这类原始文件名全程按原名处理）
- **自动恢复**：清单中存在但本地缺失的分卷（例如被 `--clean` 清理后需要重新核对）自动 `git restore` 恢复
- 按分卷序号顺序拼接，先写入 `.pdf.tmp` 临时文件；校验合并大小与分卷总大小一致
- 默认执行**输出内容重读一致性校验**：`flush` + `fsync` 后重读临时文件计算 SHA256，与写入流哈希比对
- 全部通过后使用原子替换（`os.replace`）生成正式 PDF

即使中途断电、手动终止或磁盘写入失败，也不会留下半截正式 PDF。

### 完整性判定原则（v1.0）

**只有哈希可以作为"已完成"的证据；文件大小和 API 元数据都不行。** 具体规则：

| 场景 | 默认（校验开启） | `--skip-verify` | `--clean`（无论是否 skip-verify） |
|---|---|---|---|
| 分卷齐全，输出已存在 | 分卷串联哈希 vs 输出哈希，一致才跳过 | 有历史记录且大小一致才快速跳过；**无记录直接重建，不信任同大小现有文件** | 同默认，且通过后才删分卷 |
| 分卷已清理，输出存在 | 记录 sha256 vs 输出重算哈希，一致才跳过；不一致/无哈希 → 恢复分卷重合并 | 记录大小一致即跳过 | 同默认，且顺带清理残余分卷 |
| 新合并输出 | 重读一致性校验 | 不重读（信任换速度） | **强制重读**（输出将成唯一副本） |
| 分卷组不在 Git 树清单 | **一律拒绝合并**（无法确认末卷完整），报错提示 | 同默认 | 同默认，且**绝不删除**其分卷 |
| 记录被标记 stale（上游变更/切源/diff 失败） | 先比对历史分卷集合：新清单缺历史卷 → **拒绝覆盖现有 PDF**；未缺 → 重新合并。stale 记录不作为任何跳过证据 | 同默认（**不可**凭 stale 记录快跳） | 同默认 |
| 清单仅有 `.pdf.1` 单卷 | **拒绝合并**（无法证明不存在被删除的后续分卷） | 同默认 | 同默认，且不删其分卷 |

日志中两种跳过会严格区分措辞："哈希核对一致，跳过"（真正验证过）与"按历史大小快速跳过（未哈希核对）"（仅 `--skip-verify`）。

`--skip-verify` 的语义是**显式选择信任历史记录以换取速度**，适合确认环境可靠时的重复运行；任何涉及删除分卷的操作（`--clean`）都无法跳过校验。

### `--clean` 的安全语义

- 分卷只在两种情况下被删除：本次合并成功并通过校验；或现有 PDF 与分卷串联哈希**完全一致**
- 分卷删除后组的完成状态（含 sha256）记录在状态文件中；此后每次校验模式运行都会用该哈希重算现有 PDF 核对，被篡改/损坏会自动恢复分卷重新合并
- 状态文件丢失或来自不含哈希的旧版本时，**不会信任 API 大小**，而是恢复分卷重新合并（安全优先，代价是重新下载这部分文件）

### 关于哈希校验的如实说明

所有校验（重读一致性、记录哈希核对）只能保证**本地合并与写入无误**。上游仓库未提供基准哈希，因此无法验证分卷内容本身与上游一致——如需最终确认，请打开合并后的 PDF 抽查。

## 断点恢复与中断

- 运行中按一次 `Ctrl+C`：完成当前数据块后保存状态退出；再按一次强制退出
- 再次运行 `python textbook.py` 即从断点继续；校验模式（默认）下每次运行都对每组做哈希核对，被删除/篡改的输出会自动重新合并
- 更换选择（`--reselect`）会完整重置检出、扫描与合并缓存并触发重新检出；已合并组的哈希记录作为持久缓存保留，回选旧目录时可快速核对跳过
- 同一目录同时只能运行一个实例（`china_textbook.lock`）；进程崩溃遗留的锁会在下次运行时自动回收

### 切换仓库源（`--repo` / `--use-gitee`）

- 切换前会检查新源与当前工作区的**历史兼容性**（`merge-base --is-ancestor`）：
  - 同源镜像（如 GitHub ↔ Gitee 同步仓库）：自动快进同步本地分支到新源
  - 无关仓库：**拒绝在当前工作区切换**并回滚到原源，请到新的空目录中重新运行
- 切换是**事务式**的：切换前快照 origin 与完整状态，任一步（fetch、分支探测、兼容性检查、分支同步）失败都完整回滚，**已完成组的哈希记录不会丢失**
- 切换成功后大小缓存、检出标记自动失效重建
- **哈希记录的失效规则（v1.0）**：切换前后 HEAD 不变（同源同进度镜像）则全部保留，免重合并；HEAD 变化（新源领先）则按 `git diff` 将上游已变化的组记录**标记为 stale**（保留历史分卷集合用于删卷检测），缺失分卷自动 `git restore` 后重新合并；diff 无法计算时全量 stale 标记重新核对（fail-closed，同样保留 parts）

## 常见问题

### 1. 为什么先 clone 再选择？
脚本使用 `git clone --filter=blob:none --no-checkout`，这一步只拿提交与目录树，不下载 PDF，所以能先看目录、再决定下载什么。

### 2. 选择前显示的大小从哪来？会不会触发下载？
来自 GitHub/Gitee 的 Trees API 元数据（一次 HTTP 请求获取全树大小并按 commit 缓存），全程不触碰 Git 对象，不会触发部分克隆的懒加载下载。若 API 不可用（如自定义仓库源），会如实显示"大小未知"并在下载前请你确认。

### 3. 提示"分卷不连续，缺少卷号"怎么办？
本地缺卷（Git 树中有）会自动恢复，无需手动处理；报错中注明"上游清单分卷不连续"则属上游数据问题，可到上游仓库反馈。

### 4. 合并后的 PDF 在哪里？会删分卷吗？
在原分卷文件所在目录下。默认不删除分卷；需要清理时运行 `--clean`（只删除哈希验证通过的组，对已合并目录同样生效）。

### 5. 运行中断怎么办？
直接再次运行 `python textbook.py`，程序会读取状态文件恢复进度。

### 6. 提示"无法读取 Git 树清单，停止处理"？
这是 fail-closed 保护：仓库元数据异常时宁可停止也不冒险合并出缺卷的 PDF。可运行 `git -C ChinaTextbook_Workspace fsck` 检查仓库，必要时删除工作区重新运行。

### 7. 提示"检测到另一实例正在运行"？
同目录不支持并行运行。若确认没有其他实例（如上次异常断电），锁文件会在下次运行时自动识别为陈旧锁并回收；也可手动删除 `china_textbook.lock`。

### 8. 提示"该分卷组不在 Git 树清单中，拒绝自动合并"？
工作区里存在 Git 未跟踪的本地分卷文件（如自行拷入的 `x.pdf.1`）。工具无法确认这类组的最后一卷是否完整，合并可能产出残缺 PDF，`--clean` 还会删掉唯一原件，因此默认拒绝。确认分卷齐全时请自行拼接（如 `cat x.pdf.* > x.pdf` / `copy /b`）或将文件移出工作区处理。

### 9. 提示"检测到上游删除了原有分卷 …，拒绝自动覆盖"？
上游清单相比本工具的历史记录少了曾经存在的分卷（例如末卷被上游删除）。此时 Git 清单看起来依然"连续"，但工具无法确认新清单是完整拆分还是误删的结果，因此保留现有 PDF 并报错（fail-closed）。处理方式：先到上游仓库确认该变更是否合法（如整组重新拆分、合并为更少的卷）；确认合法后**删除对应的输出 PDF 再重跑**，工具即按新清单重建；若疑似上游误删，请保留现有 PDF 并向上游反馈。

### 10. 提示"该组在 Git 清单中仅有 .pdf.1 单卷……已拒绝自动合并"？
正常的分卷拆分不会只有 1 卷（拆分本身就意味着原文件超过单文件体积限制），单卷组很可能是后续分卷被上游删除的残留，且首次运行没有历史记录可比对，因此默认拒绝。如到上游确认该文件本身就是完整 PDF，可自行将 `x.pdf.1` 重命名为 `x.pdf` 使用，并建议向上游反馈修正命名。

### 11. Windows 下颜色乱码或路径报错？
v4 已自动启用 VT 转义与 `core.longpaths`。若终端不支持彩色（如老旧环境），会自动降级为无色输出。

## 已知限制

- 上游仓库不提供文件基准哈希，所有校验仅覆盖**本地**合并与写入正确性，无法验证内容与上游一致
- **首次见到的多卷组无法识别任意"末卷缺失"**：末卷被上游删除后，Git 清单依然"连续"，本地没有任何依据能证明它原本有更多卷。工具能拦截的是——① 中间缺卷（连续性校验）；② **有历史记录**的组的删卷（历史分卷集合比对，v1.0）；③ 单卷组（默认拒绝，v1.0）。若某组的末卷在本工具**首次运行之前**就已被上游删除且剩余不止 1 卷，本地无手段检测，只能依赖上游提供基准清单/哈希才能彻底解决
- 记录被标记 stale（上游变更、切源、diff 失败）后不作为任何跳过证据，将触发重新核对或 fail-closed 拒绝，属安全设计而非故障
- `--skip-verify` 下按**历史记录**大小快速跳过，同大小的内容差异不会被发现（这是该选项的明确取舍；无记录的文件不享受此待遇——直接重建；`--clean` 不受其影响）
- Git 清单外的本地分卷组不会被自动合并（无法确认末卷完整性）；如确认齐全请自行拼接或移出工作区处理，后续版本计划提供 `--allow-local-splits` 显式开关（开启时默认禁用 `--clean`）
- API 大小信息依赖 GitHub/Gitee 接口可用性（GitHub 匿名接口每小时限 60 次，正常使用远低于此）
- 已合并组记录是跨选择的持久缓存；`--status` / `--report` 只显示当前选择范围内的条目，历史条目保留在状态文件中
- 若上游目录结构大幅调整，交互列表会随之变化，属预期行为
- 合并产物存放在 Git 工作区内（与分卷同目录），`git status` 会显示为未跟踪文件，属预期现象
- 尚未提供免 Python/Git 的打包版本（exe），已列入后续计划

## 后续可改进方向

- 合并产物迁移到 Git 工作区之外的独立输出目录（如 `ChinaTextbook_Output`），与仓库彻底解耦
- `--allow-local-splits` 显式开关：允许合并 Git 清单外的本地分卷组（默认禁用 `--clean`，风险自担）
- PyInstaller 单文件打包，降低非技术用户使用门槛
- 更细粒度的文件级选择
- 导出选择清单为文本文件
